A.1 Splunk Components — What Each One Does at Scale
+
You know these — here's the architect view
At scale, each component becomes a cluster or a service. The architect decides WHICH topology fits the business.
  • Universal Forwarder (UF) — Lightweight agent. No parsing. Sends raw data. You deploy via Deployment Server. At scale: thousands of UFs across servers, endpoints, cloud VMs.
  • Heavy Forwarder (HF) — Parses and filters before sending. Used when you need to reduce data volume or route data to different indexes. Acts as a "data filter" layer.
  • Indexer Cluster — Multiple indexers working together. Provides redundancy (replication factor) and search performance. Architect decides: How many peers? What replication factor? On-prem or cloud?
  • Search Head Cluster — Multiple search heads. Load balances user searches. Critical for large SOC teams. Architect designs: How many SHs? Deployer for config management.
  • Deployment Server — You already know this. Architect concern: What happens when you have 50,000 forwarders? Scaling, client groups, serverclasses.
  • License Master / Manager — Controls daily indexing volume. Architect must understand licensing implications of data onboarding decisions.
Exam tip
The exam will ask about "distributed search" — this is when search heads query multiple indexers. You route searches across sites. Understand: search factor vs replication factor. Search factor = how many searchable copies. Replication factor = total copies stored.
A.2 New Concepts You Need to Understand
+
  • Splunk Enterprise Security (ES) — A premium app on top of Splunk. Adds dashboards for security use cases: incident review, risk-based alerting, threat intelligence framework, asset/identity management. The exam lives in ES world.
  • Splunk SOAR (Security Orchestration, Automation, Response) — Automates responses to alerts. Think: alert fires → SOAR automatically blocks an IP, creates a ticket, and notifies the team. No human needed.
  • UEBA (User and Entity Behavior Analytics) — Detects unusual behavior by learning "normal." Example: user usually logs in from NYC at 9am — suddenly they log in from Russia at 3am. UEBA flags it.
  • CIM (Common Information Model) — A Splunk framework that normalizes data. Different firewalls call things differently. CIM says: let's all call it "src_ip" and "dest_ip." Makes searches work across all data sources.
  • Data Mesh / Data Lake — Modern architectures for storing massive amounts of security data cheaply and querying it when needed. The architect designs these.

Advanced Threat Intelligence & Analysis
5% of exam (~6 questions)
Lowest weight domain — study smart
Only 5% of the exam. Learn the concepts and vocabulary, don't memorize every detail. Focus on understanding the intelligence lifecycle and how it connects to Splunk ES.
1.1 What is Threat Intelligence?
+
Simple explanation
Threat intelligence is information about known attackers, attack methods, and indicators of compromise (IOCs). IOCs are "fingerprints" of attacks — like a suspicious IP address, a malware file hash, or a domain name used by hackers.
Analogy
Think of it like a neighborhood watch group sharing descriptions of known burglars. "Watch out for this license plate, this guy broke into 3 houses last week." Threat intel is that — but for cyber attacks.

Intelligence Sources

Open Source (OSINT)
Free, public sources. Examples: VirusTotal (malware hashes), AlienVault OTX, MITRE ATT&CK framework, government CISA advisories.
Commercial
Paid services with curated, high-quality intelligence. Examples: Recorded Future, CrowdStrike Intel, Mandiant. Better data, more reliable, but costs money.

The Threat Intelligence Lifecycle

  • Collection — Gather intel from OSINT + commercial sources
  • Curation — Filter out false positives, duplicates, expired IOCs
  • Evaluation/Confidence Scoring — How reliable is this intel? Is it from a trusted source? Recent? Specific?
  • Integration — Feed into Splunk ES as threat intelligence sources. ES uses these to automatically flag matching events.
  • Maintenance — IOCs expire! An IP that was malicious 2 years ago may be clean now. Regular cleanup is essential.
In Splunk ES
Splunk ES has a Threat Intelligence framework. You add intel sources (CSV files, TAXII feeds, REST APIs). ES automatically matches events against these IOCs and highlights them in the Incident Review dashboard.
1.2–1.3 Adversary Emulation & Threat Modeling
+
MITRE ATT&CK Framework
The most important framework for this domain. MITRE ATT&CK catalogs all known attacker techniques organized by phases (Initial Access, Execution, Persistence, etc.). An architect uses this to: (1) Map your detections to attack techniques, (2) Find gaps where you have NO detection, (3) Prioritize which detections to build.
Adversary Emulation — simple version
Red team (ethical hackers) pretend to be real attackers using real attacker techniques from threat intel. Then blue team (defenders) checks if their detections caught it. This tests if your security actually works — not just if it's configured.
  • Threat Modeling — Systematically asking "what would an attacker target?" For a bank: the wire transfer system. For a hospital: patient records. Then design defenses around those targets.
  • Intelligence-Informed Emulation — Instead of random red team exercises, use actual intel about threat actors that target YOUR industry. A healthcare org should practice defending against ransomware groups known to target hospitals.
Domain 1 — Quick Check
1. An organization wants to reduce false positives in their threat intelligence feed. Which lifecycle phase should they focus on improving?
A. Collection
B. Curation and confidence scoring
C. Integration into Splunk ES
D. Initial deployment of TAXII feeds
Curation filters out low-quality, expired, or inaccurate IOCs. Confidence scoring helps prioritize high-quality intel. These directly reduce false positives. Collection just gathers everything — more is not better without curation.
🎯 Domain 1 — Exam-Style MCQ Bank 7 Questions ▾
Score:0/0
MCQ 1.1 — A security architect wants to integrate threat intelligence into Splunk ES. Which format is MOST commonly used to share standardized threat intelligence feeds between platforms?
A. CSV flat files only
B. JSON over REST API only
C. STIX/TAXII
D. XML SOAP feeds
STIX (Structured Threat Information Expression) is the standard format for threat intel data. TAXII (Trusted Automated Exchange of Intelligence Information) is the transport protocol. Together they are the industry standard for sharing threat intelligence between platforms and into Splunk ES.
MCQ 1.2 — An IOC (IP address) was flagged as malicious 18 months ago. It appeared in no new threat reports since then. What should the architect recommend?
A. Keep it — all IOCs should be retained permanently
B. Remove or expire it — stale IOCs increase false positives and reduce intel quality
C. Move it to a separate "historical" index
D. Lower its alert severity from high to medium
IOC maintenance is a key lifecycle step. IP addresses change ownership frequently. A malicious IP from 18 months ago may now belong to a legitimate organization. Keeping stale IOCs causes false positives (alerting on legitimate traffic) and degrades analyst trust in the system. Regular expiration of aged IOCs is an architect best practice.
MCQ 1.3 — A healthcare organization wants to prioritize which attacks to defend against. What is the BEST approach to threat modeling?
A. Use generic threat models from the internet and apply them as-is
B. Focus only on external attackers since insider threats are rare
C. Research threat actors known to target healthcare, map their techniques to the organization's crown jewel assets (patient data, billing systems)
D. Model threats based on what competitors were attacked with last year
Intelligence-informed threat modeling means using real threat actor profiles relevant to YOUR industry. Healthcare orgs face specific threats — ransomware groups targeting patient data, nation-state actors targeting medical research. Crown jewels (most valuable assets) are the first modeling targets. Generic models waste resources defending against irrelevant threats.
MCQ 1.4 — What does MITRE ATT&CK provide that helps a security architect find detection gaps?
A. A list of known malware hashes to block
B. A comprehensive catalog of attacker techniques organized by attack phase, allowing architects to map existing detections and identify uncovered techniques
C. A scoring system for firewall rule effectiveness
D. Vendor-specific Splunk detection rules
MITRE ATT&CK is a matrix of every known attacker technique (e.g., Credential Dumping T1003, Lateral Movement T1021). By mapping your Splunk detection rules to these techniques, you get a visual coverage map — green = detected, red = blind spot. This gap analysis tells you exactly where to invest in new detections.
MCQ 1.5 — What is the difference between a tactical IOC and a strategic threat intelligence report?
A. Tactical IOCs (IPs, hashes, domains) are for immediate detection; strategic reports inform long-term security program decisions by describing attacker motivations and trends
B. Tactical IOCs are high confidence; strategic reports are low confidence
C. Tactical IOCs come from commercial vendors; strategic reports come from OSINT
D. There is no meaningful difference
Threat intelligence has multiple levels. Tactical = actionable indicators to feed into SIEM right now (block this IP, this hash). Operational = TTPs (how attackers operate). Strategic = trends, geopolitical context, which threat groups are active — used to plan the security program roadmap. An architect consumes all three levels.
MCQ 1.6 — During a purple team exercise, a red team uses an advanced technique that the blue team fails to detect. What is the MOST valuable outcome of this exercise?
A. Proof that the security team needs to be replaced
B. Justification to purchase more security tools
C. A specific, validated detection gap that the architect can now prioritize addressing with new Splunk detection rules
D. A report showing the organization failed its security audit
Purple team exercises are not about "passing" or "failing" — they're about finding and fixing gaps. A missed detection is valuable intelligence: you now know exactly what to build. The architect takes the output and creates new correlation rules, adds new data sources, or adjusts SOAR playbooks to close the gap. This is continuous improvement.
MCQ 1.7 — A Splunk ES architect wants to automatically enrich alerts with threat intelligence data when a known-bad IP is observed. Which Splunk ES feature enables this?
A. Splunk DB Connect
B. The Threat Intelligence Management framework in Splunk ES, which automatically matches events against loaded threat intel sources
C. A custom lookup table manually updated weekly
D. Splunk SOAR webhook integration
Splunk ES has a built-in Threat Intelligence Management framework. You load intel sources (TAXII feeds, CSV, REST API). ES automatically runs correlation searches that match incoming events against these IOCs and creates threat intel match events. This is automatic, real-time enrichment — no manual lookup needed.

Next DomainSecurity Data Mgmt
02
Security Data Management
20% of exam (~24 questions)
Most important domain
20% of the exam. This is the closest domain to what you already know (forwarders, indexers, data sources). The architect level asks: WHICH data to collect, HOW to store it, and HOW to make it all work together.
2.1–2.3 Data Sources — What to Collect and Why
+
Core concept: Signal vs. Noise
Not all data is equally useful. Some data generates thousands of events per second with few useful security signals (noise). Some data generates fewer events but every one is important (signal). The architect must choose wisely because storage costs money and too much data slows down searches.

High Value Data Sources

  • EDR (Endpoint Detection & Response) — Like CrowdStrike, Carbon Black. Records every process, file, network connection on an endpoint. HIGH SIGNAL. Better than plain Windows event logs alone.
  • Identity & Authentication Logs — Active Directory, Azure AD, Okta. Who logged in, from where, to what. Critical for detecting compromised accounts.
  • Network Flow (NetFlow/VPC Flow) — Records: "source IP talked to dest IP on port X for Y bytes." Doesn't capture content but shows WHO is talking to WHO. Great for detecting lateral movement.
  • DNS Logs — Every domain lookup. Attackers use DNS for command-and-control. Huge value for detecting malware callbacks.
  • Asset & Vulnerability Management — What assets exist? What vulnerabilities do they have? Context for prioritizing alerts. (If a critical server is vulnerable, alerts about it get higher priority.)
⚡ High Signal, High Noise
Packet Capture (PCAP) — Full network content. Extremely detailed but generates enormous data volume. Usually only for specific segments or triggered captures.
✅ High Signal, Lower Noise
EDR Process Flow — Process telemetry from endpoint agents. Better than raw Windows Event Logs because it's pre-filtered and richer context.
Exam scenario tip
If a question asks "an organization has limited storage — which data source provides the best coverage for detecting insider threats?", think: UEBA + Identity logs + EDR. They give good signal without overwhelming volume.
2.4 Non-Standard Environments: OT/ICS, Legacy Systems
+
What is OT/ICS?
OT = Operational Technology. ICS = Industrial Control Systems. These are systems controlling physical things — factory machines, power grids, water treatment plants. They often run OLD operating systems (Windows XP!) and cannot have traditional security agents installed.
  • Challenge — Can't install UF on a 20-year-old PLC (Programmable Logic Controller) running an industrial process.
  • Solution: Out-of-Band / Passive Monitoring — Use network taps or SPAN ports to passively copy network traffic. A separate sensor reads it and forwards to Splunk. No agent on the OT device itself.
  • Legacy Systems — Older systems may use syslog (old-school log forwarding). Use a syslog server as an intermediary — it receives syslog and forwards to Splunk via UF or HF.
  • Specialized Protocols — OT environments use industrial protocols like Modbus, DNP3, PROFINET. Need specialized parsers/TAs (Technology Add-ons) to understand this data.
2.5 Data Lifecycle Management — Hot, Warm, Cold
+
Analogy — Like your fridge and pantry
Hot data = in the fridge (fast access, expensive). Warm data = in the pantry (medium speed, cheaper). Cold data = in the basement freezer (slow access, very cheap). You put today's leftovers in the fridge. Last month's stuff in the freezer.
  • Hot Bucket — Most recent data. Actively being written to. Stored on fast SSDs. Most expensive per GB.
  • Warm Bucket — No longer being written to but still frequently searched. Moved from hot after a time period.
  • Cold Bucket — Older data. Rarely searched. Moved to cheaper/slower storage (HDDs, object storage like S3).
  • Frozen Bucket — Archived data. Splunk removes it from index but you can archive it. Can be thawed back if needed for forensics.
  • Data Retention Policy — How long to keep each type of data. Compliance requirements (PCI: 1 year, some regulations: 7 years) drive this. Architect must design retention that meets legal requirements within budget.
  • Data Residency — Some regulations require data to stay in a specific country/region. EU data must stay in EU (GDPR). Architect designs where indexers are physically located.
2.6 CIM — The Common Information Model
+
Why CIM matters (a lot)
You have 50 different firewalls from 10 different vendors. Cisco calls the source IP "src_addr". Palo Alto calls it "src_ip". Fortinet calls it "srcip". Without normalization, you'd need a different search for each. CIM says: map everything to a standard field name. Then ONE search works on ALL of them.
# WITHOUT CIM - need separate searches per vendor:
index=cisco_fw src_addr=192.168.1.5
index=palo_fw src_ip=192.168.1.5
index=fortinet srcip=192.168.1.5

# WITH CIM normalization - ONE search:
tag=network src=192.168.1.5
  • CIM Data Models — Predefined schemas for different data types: Network Traffic, Authentication, Endpoint, Web, etc.
  • Technology Add-ons (TAs) — Splunk apps that do the field mapping from vendor-specific fields to CIM fields. You install the Palo Alto TA, it maps Palo's fields to CIM automatically.
  • CEF (Common Event Format) — Another normalization standard, used by some SIEM tools. Similar concept to CIM but a different standard.
  • Why it matters for ES — Splunk Enterprise Security is built on CIM. ALL ES features (correlation searches, dashboards, threat intelligence matching) only work if your data is CIM-compliant.
2.7–2.8 Advanced Analytics: ML, AI, Data Mesh, Federated Search
+
  • Machine Learning (MLTK) — Splunk Machine Learning Toolkit lets you build models inside Splunk. Use cases: detect anomalous login times, forecast storage growth, cluster similar events.
  • Behavioral Analysis (UEBA) — Instead of rules ("alert if X happens"), learn what's normal and alert on deviations. Better at catching new attack types that don't match known signatures.
  • Data Mesh — Architecture where different teams own their own data domains but make data available to others. Decentralized. Reduces bottleneck on central data team. Good for large organizations.
  • Data Lake — Centralized repository for ALL data, structured and unstructured, raw format. Cheap storage (like S3). Query it when needed. Good for long-term retention of large volumes.
  • Message Bus (Kafka) — A pipeline that buffers and routes data between producers and consumers. Splunk can be a consumer. Helps handle massive data volumes without overwhelming indexers.
  • Federated Search — Search across multiple Splunk deployments or even non-Splunk data stores FROM a single search head. You don't have to move all data to one place — query it in place.
Federated Search analogy
Instead of moving all books from every library in the country to one mega-library, you use a search system that asks all libraries at once and combines results. The books (data) stay where they are.
Domain 2 — Quick Check
2. An organization ingests Windows Event Logs AND an EDR solution. Both cover endpoint activity. What is the MAIN reason to prefer EDR process telemetry over raw Windows Event Logs for security monitoring?
A. EDR is cheaper to store
B. EDR provides higher signal with richer context and lower noise
C. Windows Event Logs require a separate parser
D. EDR integrates with CIM automatically
EDR solutions like CrowdStrike or Carbon Black provide pre-filtered, enriched process telemetry. Raw Windows Event Logs generate massive volume with many low-value events. EDR gives you the signal (what matters) with less noise. This is a key architect trade-off.
3. A company stores 90 days of hot/warm data and needs to retain security logs for 3 years for compliance. What is the BEST architecture approach?
A. Increase indexer storage to hold 3 years of hot data
B. Delete data after 90 days to save costs
C. Implement storage tiering — move data to cold/frozen storage after 90 days, archive to object storage for 3 years
D. Forward all logs to a separate SIEM for long-term storage
Storage tiering is the architect's solution. Recent data on fast (expensive) storage for quick searching. Older compliance data on cheap cold/frozen storage (like S3). This balances cost and compliance requirements — a core architect decision.

MCQ 2.1 — An organization needs to monitor a factory floor with legacy PLCs that cannot have software installed. What monitoring strategy should the architect recommend?
A. Deploy Splunk Universal Forwarder on each PLC
B. Skip monitoring — OT devices are not in scope
C. Use network SPAN ports or TAPs for passive traffic capture, with OT-aware sensors forwarding to Splunk

OT-specific security tools (Claroty, Dragos) understand industrial protocols and forward normalized data to Splunk. No agent is required on the OT device, preserving availability and stability.
MCQ 2.2 — Which statement BEST describes the purpose of the Splunk CIM (Common Information Model)?
A. It compresses data to reduce storage costs
B. It normalizes field names from different data sources to a standard schema so searches work across all sources uniformly
C. It encrypts sensitive log fields in transit
D. It manages which users can access which indexes
CIM is Splunk's data normalization framework. Different vendors name fields differently. CIM standardizes them — src, dest, user, action. Splunk ES correlation searches are built on CIM fields, so data MUST be CIM-compliant for ES to work. Technology Add-ons (TAs) perform the field mapping automatically.
MCQ 2.3 — A large organization has security data in multiple Splunk deployments across 3 regions. Analysts need to search all regions from one interface without consolidating data. What should the architect implement?
A. Replicate all data to a central indexer cluster
B. Give analysts VPN access to each regional Splunk
C. Implement Splunk Federated Search to query all deployments from a single search head
D. Export all data nightly to a data warehouse
Federated Search allows a single Splunk search head to query multiple Splunk deployments (and even non-Splunk sources) without moving data. Data stays in its regional location — ideal for data residency compliance. Analysts get one unified search interface while the underlying data stays distributed.
MCQ 2.4 — An architect must balance security monitoring coverage against storage costs. Which data source combination provides the BEST signal-to-noise ratio for detecting compromised user accounts?
A. Full packet capture + all Windows Event Logs
B. Identity/authentication logs (AD, Okta) + EDR behavioral telemetry + UEBA baselines
C. Firewall logs + antivirus scan results
D. DNS logs + web proxy logs only
For compromised account detection, authentication logs show where/when/how users log in, EDR shows what those accounts do on endpoints, and UEBA learns normal behavior patterns. This combination is high-signal, lower-volume compared to full packet capture or all Windows Events. It targets the specific threat (account compromise) efficiently.
MCQ 2.5 — What is the PRIMARY purpose of a message bus (like Apache Kafka) in a Splunk security data architecture?
A. To replace Splunk indexers with cheaper infrastructure
B. To buffer, route, and decouple high-volume data streams so indexers are not overwhelmed by data spikes
C. To encrypt data before it reaches Splunk
D. To provide a backup copy of all indexed data
Message buses like Kafka sit between data producers and Splunk indexers. During traffic spikes, Kafka absorbs the burst and feeds Splunk at a consistent rate. It also enables data routing — send the same events to multiple consumers (Splunk + data lake + analytics platform). This is critical in enterprise architectures handling millions of events per second.
MCQ 2.6 — A company's EU operations must comply with GDPR data residency requirements for security logs. What architecture decision is MOST important?
A. Encrypt all EU logs with AES-256
B. Deploy Splunk indexers physically located in the EU so EU data never leaves EU jurisdiction
C. Tag EU logs with a GDPR label in Splunk
D. Give EU analysts a separate login
GDPR data residency means EU personal data must be stored and processed within the EU. This requires EU-region Splunk indexers — either on-premises in an EU datacenter or Splunk Cloud on an EU cloud region (AWS eu-west, Azure West Europe, etc.). Encryption and tagging don't satisfy residency requirements — physical location matters.
MCQ 2.7 — What distinguishes a Data Lake from traditional Splunk hot/warm/cold storage?
A. A data lake stores only structured, indexed data
B. A data lake stores massive volumes of raw, unprocessed data cheaply (e.g., S3) and is queried on-demand — ideal for long-term retention and big data analytics
C. A data lake replaces Splunk entirely
D. A data lake only stores structured database logs
Data lakes (like S3, Azure Data Lake) store everything raw and cheap. Unlike Splunk's indexed storage (which is optimized for fast search but expensive), data lakes trade search speed for massive scale and low cost. Architects use them for compliance archives and big data analytics. Splunk can query data lakes via Federated Search or SmartStore.
MCQ 2.8 — An architect wants to implement ML-based anomaly detection in Splunk. A new analyst asks why not just use rule-based alerts. What is the BEST answer?
A. ML is always more accurate than rules
B. Rules are too complex to write for experienced analysts
C. Rules only catch known attack patterns; ML detects novel or unknown attack behaviors by learning what is normal and flagging deviations — even if no rule matches
D. Rules generate too many false positives compared to ML
Rules are excellent for known attacks with defined signatures. ML/behavioral analytics catch zero-days, novel techniques, and insider threats where no rule exists. A sophisticated attacker who knows your detection rules can evade them — but anomaly detection based on behavioral baselines is much harder to evade. Architects use both: rules for speed, ML for coverage.

Advanced Incident Response & Management
10% of exam (~12 questions)
3.1 IR + ITSM: Making Security Part of the Business
+
Key concept: ITSM and ITIL
ITSM = IT Service Management. ITIL = a framework for ITSM. These govern how IT changes and incidents are managed in a business. The architect must make security incidents play well within this system — not operate as a separate island.
  • Change Management — Every change to a system should go through a formal approval process. Why does security care? Because attackers exploit systems DURING changes, and unauthorized changes may BE attacks. Splunk should alert on changes that bypass the change management process.
  • Incident Management Process — Security incidents should link to IT incident management (ServiceNow, Jira). Splunk SOAR can automatically create tickets when ES triggers a notable event.
  • Problem Management — After incidents, root cause analysis. The architect ensures forensic data is retained long enough to support post-incident analysis.
3.2–3.3 Large-Scale Incident Response & Forensics
+
Large-scale IR — what it means
A "large-scale incident" is a breach affecting hundreds of systems or the entire organization. Example: a ransomware attack encrypting thousands of servers. The response involves dozens of people, needs coordination, and happens under extreme pressure.
  • Incident Command Structure — Clear roles: Incident Commander, Technical Lead, Communications Lead, Legal, Executive stakeholders. The architect designs the PROCESS, not just the technology.
  • Communication Plan — Who gets notified when? What do you tell executives vs. technical team vs. customers? Splunk ES dashboards can provide executive-level summaries while analysts see technical details.
  • Forensics Readiness — The architect ensures: (1) Enough data is collected and retained, (2) Chain of custody for evidence, (3) Memory forensics capability (capturing RAM snapshots), (4) Splunk can search historical data for attacker activity going back months.
  • Forensic Data Requirements — Full packet capture for key segments, detailed process logs from EDR, authentication logs, netflow. Without these, you can't reconstruct what the attacker did.
Exam tip
Questions about large-scale IR often focus on: communication, coordination, forensic readiness, and how to ensure evidence integrity. Think "what does a Splunk architect need to pre-build before an incident happens?"
Domain 3 — Quick Check

Advanced Automation & Orchestration (SOAR)
10% of exam (~12 questions)
4.1–4.2 What is SOAR and Why Does it Matter?
+
SOAR = Security Orchestration, Automation, and Response
Splunk SOAR (formerly Phantom) is a platform that automates security workflows called "playbooks." When an alert fires, instead of an analyst manually investigating for 30 minutes, SOAR runs automated steps in seconds.
Analogy
Imagine you're a doctor and every time a patient's heart rate spikes, you have to manually check vitals, look at their history, call a specialist, and order tests. SOAR is like having a robot nurse that does all those steps automatically and only calls you if the robot can't handle it.
  • Playbooks — Step-by-step automated workflows. Example playbook for phishing: (1) Extract URLs from email, (2) Check URLs against threat intel, (3) If malicious → block sender, quarantine email, create ticket, notify user's manager.
  • Integrations / Connectors — SOAR connects to hundreds of tools: firewalls (block IP), ticketing (create ServiceNow ticket), email (quarantine messages), EDR (isolate host), Active Directory (disable account).
  • Architecture Constraint — 4.1 — Network design matters. If SOAR needs to call your firewall's API to block an IP, but the firewall is in a restricted network segment, the automation won't work. Architect must ensure SOAR has API access to all the tools it needs to orchestrate.

SOAR Architecture with Splunk ES

Splunk ES → notable event → Splunk SOAR → runs playbook → Firewall / EDR / AD / Ticket

ES detects the threat → SOAR responds automatically → tools take action
4.3–4.4 Autonomous SOC & AI/ML for Detection
+
Autonomous SOC — what does it mean?
An autonomous SOC handles routine security tasks without human involvement. Humans focus only on complex, novel threats. This is the goal — not replacing humans, but freeing them from repetitive work.
  • Tiered Automation — Tier 1 (known, low-risk alerts): fully automated. Tier 2 (medium confidence): automated with human approval. Tier 3 (novel threats): human-led with automation support.
  • Mean Time to Respond (MTTR) — Key metric for automation success. Manual response: 2 hours. SOAR response: 2 minutes. Autonomous SOC reduces MTTR dramatically.
  • AI/ML for Detection — Traditional rules: "alert if >10 failed logins." ML model: "alert if login pattern is statistically anomalous for this user." ML catches novel attacks that rules miss.
  • UEBA in context — User Entity Behavior Analytics learns baseline behavior and flags deviations. Key for insider threats and compromised accounts where there's no known IOC to match.

C. Build a SOAR playbook that automatically triages phishing alerts and only escalates confirmed malicious ones to analysts

SOAR automation is the architect's answer to alert volume. A well-designed playbook can automatically check URLs, verify sender reputation, look up threat intel, and auto-close false positives. Only real threats reach human analysts. This is the core value of SOAR.

Scaling Cybersecurity Defenses & DevSecOps
15% of exam (~18 questions)
5.1–5.2 DevSecOps — Security Baked Into Development
+
What is DevSecOps?
Traditional software development: Developers build the app → Security team reviews it at the end → Often finds 100 problems → Expensive to fix. DevSecOps: Security is integrated throughout the development process. Developers find security issues while coding, not after deployment.
Analogy
Building a house: old way = build the whole house, then invite a safety inspector who says "you need to tear down 3 walls." DevSecOps = safety inspector is on-site every day, issues caught when they're just nails in a stud, not full walls.
  • Detection as Code — Writing security detection rules (Splunk searches/alerts) as code, stored in version control (Git). Just like developers manage their code. Benefits: version history, code review, automated testing, can be deployed automatically.
  • CI/CD Pipeline — Continuous Integration / Continuous Deployment. Automated pipeline that tests and deploys code changes. Security: run security scans in the pipeline. If code fails security scan → deployment blocked.
  • Security Sensors in DevOps — Add security monitoring to the application build and deployment pipeline. Log every deployment, configuration change, and secret access.
# Detection as Code example — stored in Git:
name: Brute Force Detection
search: index=auth action=failure | stats count by user | where count > 10
cron: */5 * * * *
alert: email, create_notable_event
5.3 SBOM — Software Bill of Materials
+
What is an SBOM?
A complete list of all software components inside an application — including third-party libraries and open-source packages. Like a nutrition label, but for software.
Why it matters for security
In 2021, Log4Shell (a massive vulnerability in the Log4j library) affected thousands of applications. Organizations with an SBOM could immediately search "do we use Log4j?" and know in minutes. Organizations without one spent weeks figuring it out. The architect mandates SBOM generation as part of the build process.
  • Use in security response — When a new vulnerability is announced (CVE), search SBOM to find every application using the affected component. Prioritize patching those apps first.
  • Integration with Splunk — Import SBOM data into Splunk. Correlate with vulnerability scan data. When ES fires an alert on a server, enrich it with SBOM info to understand what software is at risk.
5.4–5.6 Paved Roads & Architecture Blueprints
+
Paved Roads concept
A "paved road" is a pre-built, pre-approved, security-compliant way to do something. Instead of every team building their own logging setup from scratch (and getting it wrong), the security team provides a "paved road" — a pre-configured, ready-to-use solution.
  • Example paved road — A Docker container image pre-configured with the Splunk UF and all required logging settings. Developer just uses this image → their app is automatically onboarded to Splunk with correct configs.
  • Architecture Blueprints — Standard, documented designs for common scenarios. "Here's the blueprint for adding a new cloud app to our security monitoring." Teams follow the blueprint instead of reinventing the wheel.
  • Infrastructure and App Architecture Support — Security architect must understand how the application is deployed (microservices? serverless? containers?) to know what security sensors to recommend. Different architectures need different instrumentation.

Governance, Risk, and Compliance (GRC)
10% of exam (~12 questions)
What GRC means for a security architect
GRC is about aligning security to business requirements, legal requirements, and industry standards. The architect must know which frameworks apply and how they influence architecture decisions.
6.1–6.3 Key Frameworks and Regulations
+
NIST CSF
NIST Cybersecurity Framework. 5 functions: Identify, Protect, Detect, Respond, Recover. A universal framework for organizing security controls. Architect maps the organization's security capabilities to these 5 functions to find gaps.
GDPR
EU data privacy law. Affects security logging: you can't log PII (personal data) without consent. Data must stay in EU (data residency). Must delete data on request (right to erasure). Architect must design logging systems that comply.
PCI DSS
Payment Card Industry standard. If you handle credit cards, you must comply. Requires: 12 months log retention, access control, monitoring of all access to cardholder data, annual penetration testing.
HIPAA
US healthcare data protection. Protect patient health information. Requires: audit logs of who accessed patient records, breach notification, encryption. Architect designs logging around protected health information (PHI).
GDPR impact on Splunk architecture
If you're logging user activity and it includes EU citizens' data, you might be capturing GDPR-protected information in your Splunk indexes. The architect must: (1) Design data masking for PII fields, (2) Ensure EU data stays on EU indexers, (3) Build a process to delete specific users' data on request.
  • OT/ICS Security Frameworks — NERC CIP (energy sector), IEC 62443 (industrial systems). These have specific requirements for how OT environments must be monitored and protected.
  • NIST SP 800-53 — Detailed security controls catalog. Often used by government organizations and their contractors.
6.4–6.5 Risk Management and Security Cost Justification
+
Business reality: Security costs money
The architect must speak "business language." Executives don't care about technical security details — they care about risk and cost. You must justify security spending by quantifying risk reduction.
  • Risk = Likelihood × Impact — A very likely attack with small impact may be lower priority than a rare attack with catastrophic impact (e.g., ransomware shutting down operations).
  • Risk Tolerance — Every organization has a different appetite for risk. A startup may accept more risk to move fast. A bank must be extremely risk-averse. The architect designs security aligned to the organization's risk tolerance.
  • Cost-Benefit Analysis — "This SOAR implementation costs $X per year, but will save $Y in analyst time and reduce breach probability by Z%. Expected ROI: ___." Architects must make this case.
  • Residual Risk — After all controls are in place, some risk remains. The architect must clearly communicate what residual risk exists to executives.

Measuring & Improving Security Program Effectiveness
15% of exam (~18 questions)
7.1–7.2 Security Metrics — What to Measure
+
Why metrics matter
"You can't improve what you can't measure." Security metrics tell you: Is our security getting better or worse? Are our controls actually working? Where should we invest next? The architect designs the measurement framework, not just the technical controls.

Key Security Metrics

Detection Metrics
MTTD (Mean Time to Detect) — How long until we notice an attack?

Detection Coverage — What % of MITRE ATT&CK techniques do we have detections for?

False Positive Rate — What % of alerts are not real threats?
Response Metrics
MTTR (Mean Time to Respond) — How long to contain an incident?

MTTC (Mean Time to Contain) — How long to stop the attacker?

Incident Volume Trend — Are incidents increasing or decreasing month-over-month?
Risk tolerance drives metrics
A payment processor may require MTTR under 15 minutes because every minute of breach means credit card theft. A non-critical internal system might tolerate 4-hour MTTR. The architect sets SLAs for each tier of system based on business risk tolerance.
7.3–7.4 Continuous Improvement & Purple Teaming
+
  • Purple Team — Red team (attackers) and blue team (defenders) work TOGETHER. Red team attacks, blue team tries to detect. Then they share notes: "You did X, why didn't you detect it?" Used to continuously improve detection rules.
  • Continuous Process Improvement — After every incident and every purple team exercise: document what worked and what didn't → update playbooks, detection rules, processes. The program never "finishes."
  • Control Testing — Regularly verify that controls still work. A firewall rule created 2 years ago may have been overridden. An alert that worked last year may be broken by a data format change. Architect builds systematic testing into the program.
  • Gap Remediation — When testing reveals gaps: prioritize fixes based on risk. A gap in detecting ransomware behavior is more critical than a gap in detecting low-sophistication script kiddies.
MITRE ATT&CK for gap analysis
Map all your Splunk detection rules to MITRE ATT&CK techniques. Visually see which techniques you detect (green) and which have no coverage (red). This is your gap analysis. Prioritize building detections for the techniques most used by threat actors targeting your industry.

Security Capability Selection, Placement & Configuration
15% of exam (~18 questions)
8.1–8.2 Security Capability Framework: Prevent, Detect, Respond, Recover
+
The 4 pillars of defense
No single tool does everything. The architect must ensure the organization has capabilities across ALL 4 pillars. Most organizations over-invest in prevention and under-invest in detection and response.
🛡️ Prevention
Stop attacks before they succeed. Examples: Firewall rules, EDR blocking, email filtering, patch management, MFA. The goal: reduce attack surface.
🔍 Detection
Identify attacks in progress. Examples: Splunk ES correlation rules, UEBA, IDS/IPS alerts, threat intel matching. The goal: minimize MTTD.
⚡ Response
Contain and eradicate threats. Examples: SOAR playbooks, IR procedures, isolation capabilities, forensic tools. The goal: minimize MTTR.
🔄 Recovery
Restore normal operations. Examples: Backups, DR plans, business continuity procedures. The goal: minimize downtime after an incident.
Coverage Gap Analysis
The architect maps current tools to each capability area and finds gaps. "We have great prevention controls, but if an attacker bypasses them, we have no way to detect lateral movement." That's a detection gap. Fix it with network monitoring + Splunk ES detection rules.
8.3–8.5 Selecting and Justifying Security Technologies
+
  • Business Alignment — Security tool selection must align with what the business values most. A healthcare org needs HIPAA-compliant tools and must prioritize PHI protection. A fintech prioritizes transaction integrity.
  • Technology Landscape Fit — The tool must integrate with existing systems. If everything is AWS cloud, an on-prem security tool that doesn't have cloud connectors is a bad fit.
  • Total Cost of Ownership (TCO) — License cost + implementation + staff training + ongoing maintenance. The "cheapest" tool may be most expensive when you include TCO.
  • Vendor Evaluation Criteria — Reliability, support quality, roadmap, customer references, integration ecosystem. The architect leads procurement decisions with technical input.
  • Implementation Strategy — Phased rollout vs. big bang. Pilot with a small group first. Define success metrics before deployment. Have a rollback plan.
Exam scenario tip for domain 8
Questions like: "An organization is migrating to AWS. Their current SIEM is on-premises. What should the security architect recommend?" Answer: Either migrate Splunk to cloud (Splunk Cloud) or use a hybrid architecture with cloud forwarders sending to on-prem indexers, depending on cost, latency, and compliance requirements.
8.6 Resilient Solutions — High Availability & Disaster Recovery
+
Resilience in security infrastructure
The security monitoring system itself must be resilient. If Splunk goes down during a major attack, you're blind exactly when you need visibility most. The architect designs for HA (High Availability) and DR (Disaster Recovery).
  • Indexer Clustering — Multiple indexers with replication. If one indexer fails, data is not lost and search continues on remaining indexers.
  • Search Head Clustering — Multiple search heads. If one fails, users fail over to another. No single point of failure for analysts.
  • Forwarder Redundancy — Configure UFs with multiple indexers in the outputs.conf. If primary indexer is unreachable, forwarder automatically routes to backup.
  • Multi-Site Clustering — Data replicated across two physical data centers. If entire site fails, the other site has all data and can continue operations.
  • RTO and RPO — Recovery Time Objective (how fast must we recover?) and Recovery Point Objective (how much data loss is acceptable?). Architect designs infrastructure to meet these targets.

D. Ignore it — Kerberoasting is an old technique
Prioritization based on risk is the architect's approach. Consider: (1) Threat relevance — is Kerberoasting used by APT groups targeting your industry? (2) Impact — Kerberoasting leads to privileged account compromise, which is critical. (3) Compensating controls — do you have any other detection for the downstream effects? If risk is high and no compensating controls exist, prioritize it. This is risk-based decision making.

📖 Key Terms Glossary
  • APT (Advanced Persistent Threat) — Sophisticated, long-term attack, often nation-state sponsored. Stays hidden in a network for months.
  • CI/CD — Continuous Integration / Continuous Deployment. Automated pipeline for building, testing, and deploying software.
  • CIM — Common Information Model. Splunk's data normalization framework. Maps vendor-specific fields to standard names.
  • CVE — Common Vulnerability and Exposure. Standardized identifier for software vulnerabilities. (e.g., CVE-2021-44228 = Log4Shell)
  • Data Mesh — Decentralized data architecture where domain teams own and share their data.
  • EDR — Endpoint Detection & Response. Security software on endpoints providing rich telemetry and response capabilities.
  • Federated Search — Searching across multiple Splunk instances or data sources from a single interface.
  • GRC — Governance, Risk, and Compliance. Framework for aligning security to business, legal, and regulatory requirements.
  • IOC — Indicator of Compromise. Digital fingerprint of an attack: malicious IP, file hash, domain, registry key.
  • ITSM/ITIL — IT Service Management / IT Infrastructure Library. Framework for managing IT services and changes.
  • Kerberoasting — Attack technique targeting Active Directory service accounts. Attacker requests service tickets, then cracks them offline.
  • Lateral Movement — Attacker moving from one compromised system to others within the network to find high-value targets.
  • MITRE ATT&CK — Framework cataloging all known attacker techniques, organized by attack phase.
  • MTTD — Mean Time to Detect. Average time from attack start to detection. Lower = better.
  • MTTR — Mean Time to Respond. Average time from detection to containment. Lower = better.
  • OT/ICS — Operational Technology / Industrial Control Systems. Systems controlling physical infrastructure.
  • PHI — Protected Health Information. Patient medical data protected by HIPAA.
  • PII — Personally Identifiable Information. Data that identifies a person. Protected by GDPR and other laws.
  • Purple Team — Red team (attackers) and blue team (defenders) working together to test and improve defenses.
  • RBA — Risk-Based Alerting. Splunk ES feature that aggregates risk scores to reduce false positives.
  • SBOM — Software Bill of Materials. Complete inventory of all components in a software application.
  • SOAR — Security Orchestration, Automation, and Response. Platform for automating security workflows.
  • SOC — Security Operations Center. Team and technology for monitoring and responding to security threats.
  • SPAN Port — Switch Port Analyzer. Copies network traffic to a monitoring device without interrupting the original traffic.
  • TAXII — Trusted Automated Exchange of Intelligence Information. Protocol for sharing threat intelligence feeds.
  • TCO — Total Cost of Ownership. Full cost of a technology including purchase, implementation, operation, and maintenance.
  • UEBA — User and Entity Behavior Analytics. Detects threats by finding deviations from normal behavior patterns.
  • Zero Trust — Security model: trust nothing by default, verify everything. No implicit trust based on network location.
